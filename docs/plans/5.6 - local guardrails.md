### What was implemented

- Added `backend/providers/guardrail_provider.py` with:
  - `GuardrailProvider` protocol.
  - `GuardrailResult` normalized action/category/confidence/provider/model metadata.
  - `OllamaGuardrailProvider` for local Ollama model checks with `llama-guard3:1b`.
  - `safe_guardrail_check` for strict-mode fail-closed and non-strict fail-open behavior.
- Added backend settings and Compose wiring for:
  - `GUARDRAIL_MODEL=llama-guard3:1b`
  - timeout/retry flags
  - input/transcript/context/output enable flags
  - `GUARDRAIL_STRICT_MODE=false`
- Added transcript guardrails to `ProcessingPipelineService` after transcription and before analysis.
- Persisted transcript guardrail metadata under `source.guardrails.transcript` and job `providerMetadata.guardrails.transcript`.
- Added guardrail warnings into `quality.warnings` without storing raw provider prompts or raw guardrail responses.
- Added chat input guardrails before retrieval and answer generation.
- Added retrieved-context guardrails before LLM answer generation.
- Added output guardrails before assistant message persistence.
- Added safe blocked responses and `not_enough_evidence` downgrade for unsupported answers.
- Added chat metadata with guardrail provider/model/action/categories/confidence/latency and decision counts.

### What was changed from original plan

- Guardrail provider selection was removed. Guardrails are local Ollama-backed only; strict mode controls whether provider outages fail open or fail closed.
- Ollama now runs inside Compose and is reached by backend/worker through `OLLAMA_BASE_URL=http://ollama:11434`.
