# Phase 5.5 - Voice Processing And Rerank

## Status: Done

## Objectives

1. Replace placeholder audio transcription with a real voice pipeline for uploaded audio/video files and browser recordings.
2. Produce the same `TranscriptSegment[]` contract for voice and text inputs so Phase 4/5 processing, JSON generation, chunking, Milvus indexing, and chat stay shared.
3. Add speaker diarization with local speaker embeddings, using WeSpeaker as the preferred speaker embedding path.
4. Add a local Ollama-backed reranker to improve RAG ordering before answer generation.

### What was implemented

- Phase 5.5 is complete. The implemented scope includes:
  - Voice provider contracts for audio preprocessing, VAD, ASR, and diarization.
  - ffmpeg-backed local audio preprocessing that normalizes supported audio/video assets to 16 kHz mono WAV in a worker temp directory.
  - Stable derived WAV output paths per asset so worker retries reuse the same local normalized audio instead of creating duplicate files.
  - WAV metadata extraction for duration, sample rate, channel count, and preprocessing warnings.
  - Local energy-based VAD that emits speech regions and merges nearby speech windows before the ASR adapter boundary.
  - Safe voice failure handling when preprocessing, VAD, ASR, or diarization providers fail.
  - Voice warning propagation into `quality.warnings`, `source.voiceMetadata`, and processing job provider metadata.
  - Local ASR command adapter backed by `backend.model_runners.asr`, using `faster-whisper` CPU `int8` over the downloaded `/models/asr` snapshot.
  - WeSpeaker-oriented local diarization command provider backed by `backend.model_runners.diarization`, using the downloaded `/models/diarization` snapshot and merging speaker assignments into ASR transcript segments.
  - `LocalTranscriptionProvider` now routes text uploads through text extraction and audio assets through the voice provider contract; voice failures raise a safe processing error instead of creating placeholder transcript text.
  - Voice/transcription metadata is written into `meeting-intelligence-result.v1.source.voiceMetadata` and processing job provider metadata.
  - Local command rerank provider backed by `backend.model_runners.rerank`, using SentenceTransformers `CrossEncoder` over the downloaded `/models/rerank` snapshot.
  - Retrieval now collects a larger candidate set and reranks before chat answer generation.
  - Chat assistant metadata records rerank provider/model/input/output counts without storing prompts.
  - `.env`, `.env.example`, and Compose backend/worker environment wiring include Phase 5.5 model variables, including Ollama URL, embedding model, ffmpeg work-dir, ASR/diarization commands, and VAD tuning variables.
  - `ollama-init` pulls `qwen2.5:1.5b`, `nomic-embed-text`, and `llama-guard3:1b` into the Compose `ollama_data` volume.
  - `model-init` downloads default Hugging Face snapshots into the Compose `model_cache` volume: `Systran/faster-whisper-small`, `Wespeaker/wespeaker-voxceleb-resnet34-LM`, and `BAAI/bge-reranker-v2-m3`.

### What was changed from original plan

- ASR, speaker embedding/diarization, text embedding, rerank, and guardrail model paths are local-only. Only the LLM provider may use API/private endpoint paths.
- Text embedding is now model-backed through local Ollama (`nomic-embed-text`) rather than a production hash fallback.
- Concrete ASR and diarization runtime paths are now default Compose configuration through `ASR_COMMAND` and `DIARIZATION_COMMAND`; the command templates call repository-owned Python runner modules instead of requiring a separate external wrapper.
- Rerank is model-backed through `RERANK_COMMAND`; when it is not configured, retrieval proceeds without pretending a rerank occurred.
